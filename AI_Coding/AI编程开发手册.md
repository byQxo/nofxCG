第1章 总则与核心原则

1.1 先理解，后动手

所有编码工作开始前，AI 必须先完整阅读并理解项目的架构、目录结构、技术栈和业务逻辑。禁止在未掌握项目全貌的情况下直接修改代码。



1.2 最小变更原则

每次修改只做必要的变更，不引入无关改动。修改代码时严禁创建新文件，除非经过明确许可。



1.3 上下文持久化

项目的架构分析、技术选型、核心设计决策必须持久化保存（如 .kiro/steering/、.cursorrules、AGENTS.md 等），确保每次会话都能快速恢复上下文。



1.4 渐进式披露

面对大型项目时，采用渐进式探索策略。先了解整体架构，再逐步深入各模块细节，避免一次性加载过多上下文导致信息过载。



1.5 中文优先

代码注释、提交信息、文档输出统一使用中文，确保团队沟通效率和文档可读性。

第5章 数据库与数据模型设计

5.1 设计规则

表名和字段名统一使用 snake\_case 命名规范

每张表必须包含：id（主键）、created\_at（创建时间）、updated\_at（更新时间）

5.2 主数据设计原则

核心业务实体采用逻辑删除（soft delete），使用 deleted\_at 字段标记

高频查询字段必须建立索引，复合查询建立联合索引

大文本字段（如 JSON、富文本）考虑独立存储，避免影响主表查询性能

第7章 后端开发规则

7.1 代码组织规则

严格遵循项目已有的目录结构和分层架构

Controller / Service / Repository（DAO）职责分离，禁止跨层调用

公共工具类和常量统一放置在 common / utils 目录

配置项统一管理，禁止在代码中硬编码配置值

7.2 编码规范

方法长度不超过 80 行，超过则拆分为子方法

所有公共方法必须添加中文注释，说明功能、参数和返回值

异常处理必须具体化，禁止使用空的 catch 块

数据库操作必须使用参数化查询，防止 SQL 注入

敏感信息（密码、密钥等）禁止明文存储，必须加密或使用环境变量

7.3 修改代码的铁律

铁律 - 必须严格遵守

修改前必须阅读并理解相关代码的完整上下文

只修改必要的部分，不做任何"顺手优化"

修改后必须确保所有现有功能不受影响

每次修改必须能清晰说明改了什么、为什么改

禁止删除看似无用的代码，除非已确认其确实无引用

第8章 前端开发与 UI 设计规则

8.1 UI 设计原则

保持与项目现有 UI 风格一致，包括配色、字体、间距、圆角等

优先使用项目已引入的 UI 组件库，避免自定义样式

响应式设计：所有页面必须适配桌面端和移动端

表单设计必须包含输入校验和错误提示

列表页面必须支持分页、搜索和排序

操作反馈：所有用户操作必须有明确的成功/失败提示

加载状态：异步操作必须显示 loading 状态

8.2 交互设计规则

危险操作（删除、重置等）必须有二次确认弹窗

表单提交后禁用按钮，防止重复提交

长列表使用虚拟滚动或分页加载

错误信息必须友好且具有指导性，避免展示技术细节

支持键盘操作和无障碍访问（Accessibility）

8.3 前端开发规范

组件拆分粒度适中，单个组件不超过 300 行

状态管理遵循项目约定（Vuex/Pinia/Redux 等）

API 调用统一封装，禁止在组件中直接使用 fetch/axios

样式使用 scoped 或 CSS Modules，避免全局污染

第9章 开发过程管理

9.1 任务拆分规则

将大需求拆分为独立的功能模块

每个模块再拆分为具体的开发任务

每个任务的工作量控制在单次对话可完成的范围内

任务之间明确依赖关系和执行顺序

9.2 单次输出控制

单次代码输出不超过 200 行，超过则分批输出

每次输出后确认代码正确，再继续下一部分

复杂功能采用"骨架先行"策略：先输出整体结构，再逐步填充细节

9.3 变更追踪规则

每次修改必须记录：修改的文件、修改的内容、修改的原因

使用变更日志（CHANGELOG）记录重要变更

代码审查时必须能追溯每个变更的来源和目的

9.4 防止上下文丢失

长对话中定期总结已完成的工作和待办事项

关键决策和设计变更及时更新到持久化文件

切换任务前，确保当前任务的状态已保存

使用检查点机制，每完成一个里程碑就提交代码

9.5 Bug 修复规则

复现问题：首先确认 Bug 的复现步骤和条件

定位根因：通过日志、调试工具定位问题根本原因，而非表面现象

最小修复：只修改导致 Bug 的代码，不做额外改动

回归测试：修复后验证原有功能不受影响

记录总结：记录 Bug 原因和修复方案，避免同类问题再次出现

第14章 Git 版本管理与团队协作

14.1 分支管理规范

推荐分支模型：



main (生产分支)

&nbsp;├── develop (开发主分支)

&nbsp;│    ├── feature/user-auth (功能分支)

&nbsp;│    ├── feature/order-module (功能分支)

&nbsp;│    └── feature/payment (功能分支)

&nbsp;├── release/v1.2.0 (发布分支)

&nbsp;├── hotfix/fix-login-bug (热修复分支)

&nbsp;└── bugfix/fix-pagination (缺陷修复分支)

分支命名规则：



feature/功能名称 - 新功能开发

bugfix/问题描述 - 缺陷修复

hotfix/紧急问题 - 生产环境紧急修复

release/版本号 - 发布准备

14.2 提交信息规范

提交信息格式：





<type>(<scope>): <subject>



// 示例

feat(user): 新增用户注册功能

fix(order): 修复订单金额计算错误

refactor(auth): 重构登录鉴权逻辑

Type 类型定义：



类型	说明

feat	新功能

fix	Bug 修复

refactor	代码重构（不影响功能）

style	代码格式调整（不影响逻辑）

docs	文档更新

chore	构建工具或辅助工具变更

perf	性能优化

14.3 AI 编程场景下的 Git 规则

每完成一个独立功能点就提交一次，禁止大量代码一次性提交

提交前必须运行测试，确保所有测试通过

AI 生成的代码必须经过人工审查后再提交

提交信息必须准确描述变更内容，禁止使用"更新代码"等模糊描述

多人协作时，每次开始工作前先拉取最新代码，避免冲突
